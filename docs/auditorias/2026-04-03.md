## AGENTE 1 — Arquitetura e Engenharia de Sistemas

**Nota:** 5.5/10
**Status:** ⚠️ Atenção

### Escopo Analisado
App.tsx, estrutura de pastas, package.json, AuthContext.tsx, lib/supabase.ts, rotas, contextos, padrão de portais.

### Achados Principais

- **ARCH-001** 🔴 — `// @ts-nocheck` em 507 de ~498 arquivos TS/TSX
  - **Evidência:** `grep -r "@ts-nocheck" src/ | wc -l` → 507
  - **Impacto:** TypeScript desativado funcionalmente; bugs de tipo silenciosos; refactoring perigoso
  - **Risco se não corrigir:** Regressões invisíveis a cada mudança; impossível garantir contratos de dados
  - **Resultado esperado:** Após remoção gradual, type-safety real; IDE ajuda a detectar bugs antes de produção
  - **Esforço:** Alto
  - **SLA:** 30d

- **ARCH-002** 🟡 — Dual Supabase client (lib/supabase + integrations/supabase/client)
  - **Evidência:** AuthContext.tsx importa de ambos; componentes usam um ou outro indistintamente
  - **Impacto:** Sessions e tokens podem divergir; race conditions de auth; comportamento imprevisível multi-portal
  - **Risco se não corrigir:** Usuário autenticado em um client mas não no outro; APIs chamadas sem token
  - **Resultado esperado:** Um único client canônico, comportamento previsível em todos os portais
  - **Esforço:** Médio
  - **SLA:** 7d

- **ARCH-003** 🟡 — Separação de portais via `window.location.pathname` em runtime
  - **Evidência:** lib/supabase.ts: `storageKey = pathname.includes('/acessofranquia') ? 'noe-franchise-auth' : 'noe-saas-auth'`
  - **Impacto:** Frágil; quebra com mudanças de rota; dificulta testes; lógica de negócio em infraestrutura
  - **Risco se não corrigir:** Cross-portal session leak se rota mudar; impossível testar em SSR ou sem DOM
  - **Resultado esperado:** Separação por subdomínio ou env var; client determinístico
  - **Esforço:** Médio
  - **SLA:** 30d

- **ARCH-004** 🟠 — Monolito frontend de 218k linhas sem separação de domínios
  - **Evidência:** src/pages/ mistura Franqueadora, Franqueado e SaaS sem separação física
  - **Impacto:** Qualquer mudança pode afetar os 3 portais; bundle sharing desnecessário
  - **Risco se não corrigir:** Feature de um portal vaza para outro; performance degradada por bundle compartilhado
  - **Resultado esperado:** Separação por domínio (feature folders) com shared/ explícito
  - **Esforço:** Alto
  - **SLA:** 90d

- **ARCH-005** 🟠 — Lógica de negócio espalhada em componentes (sem service layer)
  - **Evidência:** Regras de cotação, crédito, automações diretamente em hooks e componentes
  - **Impacto:** Duplicação; impossível reusar lógica; testes difíceis
  - **Risco se não corrigir:** Lógica de crédito divergente entre portais; bugs de negócio difíceis de rastrear
  - **Resultado esperado:** Services/use-cases isolados, hooks apenas orquestram UI
  - **Esforço:** Alto
  - **SLA:** 90d

- **ARCH-006** 🟢 — Lazy loading robusto (490 imports dinâmicos, single Suspense boundary)
  - **Evidência:** App.tsx usa React.lazy para todas as 40+ páginas; PageLoader consistente
  - **Impacto positivo:** Bundle inicial mínimo; carregamento sob demanda funcional
  - **SLA:** Manter

### Cruzamento Doc vs Implementação
- Documentado: 52 edge functions → Real: 70 (+18 não documentadas)
- Documentado: 114 migrations → Real: 130 (+16 não documentadas)
- Não documentado: Dual Supabase client pattern, portal isolation via pathname
- Divergência: Docs mencionam "multi-tenant RLS robusto" — confirmado no banco, mas AuthContext tem fallback hardcoded que quebra o modelo

### Comparação com Auditoria Anterior
Primeira execução — sem comparação (BASELINE).

### Recomendações
- **Imediatas (24h-7d):** Consolidar para um único Supabase client canônico (ARCH-002)
- **Estruturais (30d):** Remover @ts-nocheck progressivamente começando por domínios críticos (auth, pagamentos)
- **Evolutivas (90d+):** Separação física de portais; service layer de domínio

## AGENTE 4 — Backend, APIs e Regras de Negócio

**Nota:** 6.2/10
**Status:** ⚠️ Atenção

### Escopo Analisado
70 Edge Functions, padrões de resposta, validação, idempotência, regras de negócio, CORS.

### Achados Principais

- **API-001** 🔴 — `jsonOk()` em asaas-webhook referencia `req` fora do handler scope → ReferenceError
  - **Evidência:** `supabase/functions/asaas-webhook/index.ts`: helper `jsonOk()` definido fora de `Deno.serve()`, usa `req` implicitamente fora de escopo
  - **Impacto:** TODOS os pagamentos bem-sucedidos falham com 500; Asaas tenta retry → possível duplo crédito quando sistema é corrigido
  - **Risco se não corrigir:** Sistema de pagamentos quebrado em produção; perda de receita; risco de créditos duplicados
  - **Resultado esperado:** Payments processados; créditos creditados corretamente; zero retries por erro de escopo
  - **Esforço:** Baixo (mover helper para dentro do handler)
  - **SLA:** 24h

- **API-002** 🔴 — `jsonRes()` em google-calendar-sync: mesmo bug de escopo → calendar sync quebrado
  - **Evidência:** `supabase/functions/google-calendar-sync/index.ts` — mesmo padrão
  - **Impacto:** Sincronização com Google Calendar completamente não funcional em produção
  - **Risco se não corrigir:** Reuniões não sincronizadas; feature vendida como funcional mas quebrada
  - **Resultado esperado:** Sync funcional; eventos criados no Google Calendar corretamente
  - **Esforço:** Baixo
  - **SLA:** 24h

- **API-003** 🔴 — signup-saas: modo legacy aceita `body.user_id` sem validar JWT → IDOR
  - **Evidência:** `supabase/functions/signup-saas/index.ts`: se `body.user_id` presente, usa sem verificar JWT do caller
  - **Impacto:** Atacante pode forçar criação de organização em nome de qualquer user_id existente
  - **Risco se não corrigir:** Account takeover; criação de orgs fraudulentas; violação de isolamento multi-tenant
  - **Resultado esperado:** user_id sempre derivado do JWT; body.user_id ignorado ou validado contra JWT
  - **Esforço:** Baixo
  - **SLA:** 24h

- **API-004** 🟡 — CORS com `Allow-Origin: *` e todos os métodos incluindo DELETE/PUT
  - **Evidência:** `_shared/cors.ts`: wildcard origin + `Access-Control-Allow-Methods: DELETE, POST, PUT, GET, OPTIONS`
  - **Impacto:** APIs podem ser chamadas de qualquer domínio; aumenta superfície de ataque CSRF
  - **Risco se não corrigir:** Requests maliciosos de origens externas; CSRF em endpoints autenticados
  - **Resultado esperado:** Allowlist de origens legítimas; CORS restritivo por ambiente
  - **Esforço:** Baixo
  - **SLA:** 7d

- **API-005** 🟡 — crm-run-automations: marca eventos `processed=true` mesmo em falha; sem DLQ
  - **Evidência:** `supabase/functions/crm-run-automations/index.ts`: UPDATE processed=true no início do processamento
  - **Impacto:** Automações falham silenciosamente; não há reprocessamento; eventos perdidos
  - **Risco se não corrigir:** Automações de CRM (follow-up, atribuição) perdem eventos definitivamente
  - **Resultado esperado:** Transação atômica; processed=true apenas em sucesso; error_count com retry limit
  - **Esforço:** Médio
  - **SLA:** 7d

- **API-006** 🟠 — Round-robin sem SELECT FOR UPDATE → race condition em atribuição de leads
  - **Evidência:** crm-run-automations: múltiplas funções executando simultaneamente podem atribuir o mesmo lead a múltiplos agentes
  - **Impacto:** Leads duplicados em múltiplos agentes; conflito de ownership
  - **Risco se não corrigir:** Agentes de venda disputando o mesmo lead; experiência do cliente degradada
  - **Resultado esperado:** Advisory locks ou SELECT FOR UPDATE; atribuição atômica garantida
  - **Esforço:** Médio
  - **SLA:** 7d

- **API-007** 🟠 — 23 edge functions com `verify_jwt=false` sem autenticação manual compensatória
  - **Evidência:** supabase/config.toml: 23 entradas verify_jwt=false
  - **Impacto:** Endpoints públicos que deveriam ser protegidos; dados sensíveis expostos
  - **Risco se não corrigir:** Acesso não autorizado a dados de seguradoras, orgs, usuários
  - **Resultado esperado:** verify_jwt=true para todas as funções que não são webhooks; webhooks com HMAC
  - **Esforço:** Médio
  - **SLA:** 7d

## AGENTE 6 — Segurança e Confiabilidade

**Nota:** 6.2/10
**Status:** ⚠️ Atenção — Bugs críticos pontuais em superfície ampla

### Escopo Analisado
.env no git, secrets, verify_jwt, service_role, CORS, webhooks, IDOR, RLS, HMAC.

### Achados Principais

- **SEC-NOE-001** 🔴 — .env commitado no histórico git (não está em .gitignore explicitamente)
  - **Evidência:** `.gitignore` tem `*.local` mas não `.env`; `git log --all --name-only | grep "^\.env$"` confirma presença
  - **Impacto:** VITE_SUPABASE_PUBLISHABLE_KEY exposta; histórico git acessível por todos com acesso ao repo
  - **Risco se não corrigir:** Qualquer pessoa com acesso ao repo pode usar credenciais do Supabase
  - **Resultado esperado:** `.env` em `.gitignore`; rotação de credentials; BFG para limpar histórico
  - **Esforço:** Baixo
  - **SLA:** 24h

- **SEC-NOE-002** 🔴 — 23 edge functions com verify_jwt=false (inclui funções não-webhook)
  - **Evidência:** supabase/config.toml: apenas agent-followup-cron deveria ter verify_jwt=false
  - **Impacto:** APIs sensíveis acessíveis sem autenticação; dados de múltiplas organizações expostos
  - **Risco se não corrigir:** Acesso não autorizado a dados de CRM, leads, financeiro
  - **Resultado esperado:** verify_jwt=true como default; apenas cron/webhooks externos com false + HMAC manual
  - **Esforço:** Médio
  - **SLA:** 24h

- **SEC-NOE-003** 🔴 — crm-lead-webhook: zero autenticação → qualquer POST cria lead
  - **Evidência:** `supabase/functions/crm-lead-webhook/index.ts`: sem header check, sem JWT, sem HMAC
  - **Impacto:** Flood attack → banco cheio de leads falsos; org_id válido = dados poluídos
  - **Risco se não corrigir:** DoS via lead flooding; CRM inutilizável; custo de storage crescente
  - **Resultado esperado:** HMAC obrigatório; rate limiting por org; validação de origem
  - **Esforço:** Baixo
  - **SLA:** 24h

- **SEC-NOE-004** 🔴 — Asaas webhook token opcional: `if (asaasToken) { validate } else { accept all }`
  - **Evidência:** asaas-webhook/index.ts: validação condicional; se env var não configurada, aceita qualquer request

  - **Risco se não corrigir:** Churn por experiência ruim; desconfiança no produto de IA
  - **Resultado esperado:** Script completo em primeiro intento e em retry
  - **Esforço:** Baixo
  - **SLA:** 24h

- **AI-003** 🟡 — Sem cache de respostas de IA para prompts similares
  - **Evidência:** Cada chamada de IA é nova requisição ao gateway
  - **Impacto:** Custo 2-3x maior do que necessário; latência maior para usuário
  - **Risco se não corrigir:** COGS de IA cresce linearmente com MAU sem correspondente receita
  - **Resultado esperado:** Cache semântico ou hash-based para prompts determinísticos; TTL 24h
  - **Esforço:** Médio
  - **SLA:** 30d

- **AI-004** 🟢 — BANT prompts bem estruturados nos agentes de CRM
  - **Evidência:** agent-followup-cron tem prompts com Budget, Authority, Need, Timeline explícitos
  - **Impacto positivo:** Qualidade de follow-up superior; personalização por contexto do lead
  - **SLA:** Manter e evoluir

- **AI-005** 🟢 — Gateway unificado (ai.gateway.lovable.dev) com controle de modelos
  - **Evidência:** Todas as chamadas de IA passam pelo mesmo endpoint; Gemini configurável
  - **Impacto positivo:** Troca de modelo sem alteração de código; custo controlável por gateway
  - **SLA:** Manter

- **AI-006** 🟠 — IA de geração de conteúdo sem validação de output (injection risk)
  - **Evidência:** Output de IA inserido diretamente em componentes sem sanitização
  - **Impacto:** Conteúdo malicioso em output de IA pode causar XSS se renderizado como HTML
  - **Risco se não corrigir:** Prompt injection via dados de usuário → XSS no frontend
  - **Resultado esperado:** Sanitização de HTML em todo output de IA exibido ao usuário
  - **Esforço:** Baixo
  - **SLA:** 7d
