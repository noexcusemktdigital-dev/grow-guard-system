## Padrões observados

1. **service_role full access** — padrão Lovable para tabelas de infra: `dsr_requests`, `idempotency_keys`, `webhook_events`, `meta_data_deletion_requests` usam `USING (auth.role() = 'service_role')`. Correto para acesso server-side only.

2. **Tenant isolation via `organization_memberships`** — padrão dominante nas tabelas de domínio. Policies filtram por `organization_id IN (SELECT organization_id FROM organization_memberships WHERE user_id = auth.uid())`. Cobre 63 das 160 policies explícitas.

3. **`auth.uid()` direto** — usado em tabelas de perfil (`profiles`, `user_roles`). Policy simples `WHERE id = auth.uid()`.

4. **Storage RLS** — `storage.objects` tem 12 policies cobrindo SELECT/INSERT/UPDATE/DELETE para buckets específicos.

5. **FOR ALL sem granularidade** — 51 policies usam `FOR ALL`, o que cobre todas as operações num único bloco. Comum em tabelas de lookup/configuração onde a distinção entre operações não é crítica.

---

## Lacunas

1. **7 tabelas sem RLS** — especialmente `ads_connections` e `ads_oauth_states` são alto risco: contêm tokens OAuth e segredos de plataformas de anúncios sem isolamento de tenant.

2. **76 tabelas com RLS habilitado mas sem policy** — comportamento varia conforme o caller:
   - Via `anon`/`authenticated`: **bloqueio total** (pode ser bug silencioso se frontend tenta acessar)
   - Via `service_role` (edge functions): **ignora RLS** (funciona normalmente)
   - Requer validação se acesso via frontend existe para alguma dessas tabelas.

3. **Ausência de DELETE explícito** — a maioria das tabelas cobre SELECT/INSERT mas não DELETE. Se um usuário autenticado pode deletar registros de outros tenants não é claro sem testar.

4. **`franqueado_prospections` e `franqueado_strategies`** — tabelas de dados comerciais sensíveis criadas na mesma migration sem RLS. Risco de BOLA (vazamento entre franqueados).

5. **`calendar_event_invites`** — tabela de convites sem RLS, enquanto `calendar_events` e `calendars` têm RLS habilitado. Inconsistência no módulo de calendário.

---

## Recomendações

1. **Habilitar RLS em `ads_connections` e `ads_oauth_states`** — adicionar policy `USING (user_id = auth.uid())` ou `USING (organization_id IN (...))` conforme o schema. Prioridade P0.

2. **Habilitar RLS em `franqueado_prospections` e `franqueado_strategies`** — adicionar tenant isolation por `organization_id`. Prioridade P0.

3. **Habilitar RLS em `calendar_event_invites`** — alinhar com o padrão dos módulos `calendars` e `calendar_events`. Prioridade P1.

4. **Auditar as 76 tabelas com RLS sem policy** — verificar se o frontend acessa alguma diretamente. Candidatos prioritários: `client_payments`, `finance_closings`, `saas_commissions`, `whatsapp_messages`, `team_chat_messages`.

5. **Padronizar DELETE policies** — para tabelas que permitem exclusão via frontend, adicionar policy `FOR DELETE USING (organization_id IN (...))` explicitamente.

6. **Validar `noe_service_catalog`** — se for catálogo público, pode permanecer sem RLS ou receber policy `FOR SELECT USING (true)` explícita para clareza de intenção.

## Procedimento de validação manual

```sql
-- 1. Verificar acesso cross-tenant (login como user do tenant A, tentar ver dados do tenant B)
-- Executar com role de usuário autenticado comum (não service_role)

SET LOCAL role TO 'authenticated';
SET LOCAL request.jwt.claim.sub TO '<user_id_tenant_A>';

-- Esperado: 0 linhas se RLS está correto
SELECT count(*) FROM public.franqueado_prospections
WHERE organization_id = '<org_id_tenant_B>';

-- 2. Verificar tabelas sem RLS (devem retornar dados de qualquer tenant)
SELECT count(*) FROM public.ads_connections;
-- Se retornar > 0 sem filtro: vulnerabilidade confirmada

-- 3. Verificar tabelas com RLS mas sem policy (devem retornar 0 para authenticated)
SELECT count(*) FROM public.whatsapp_messages;
-- Se retornar 0: bloqueio correto (mas pode ser bug se frontend precisa acessar)

-- 4. Checar via information_schema
SELECT tablename, rowsecurity
FROM pg_tables
WHERE schemaname = 'public'
ORDER BY tablename;

-- 5. Listar policies ativas
SELECT schemaname, tablename, policyname, roles, cmd, qual
FROM pg_policies
WHERE schemaname = 'public'
ORDER BY tablename, cmd;
```
